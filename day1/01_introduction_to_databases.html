Day 1: Introduction to Databases and SQL

Introduction to Databases and SQL

Relational Database Concepts

Databases are organized collections of structured information or data, typically stored electronically in a computer system. A relational database, specifically, organizes data into tables (or relations) with rows and columns. Each row represents a unique record, and each column represents a field in the record. This model is the foundation for most modern database systems used in business applications.

Key relational database concepts include:

  • Schema: The blueprint or structure of a database that defines how data is organized and how relationships between data are handled.
  • Tables: The fundamental structure in a relational database. Tables consist of rows and columns, similar to spreadsheets. They represent entities (like customers, products, orders).
  • Rows (Records/Tuples): Each row in a table represents a unique instance of the entity the table represents (e.g., a specific customer, a particular product).
  • Columns (Fields/Attributes): Each column represents a specific characteristic or piece of data about the entity (e.g., customer name, product price, order date).
  • Data Types: Each column is defined with a specific data type (e.g., INTEGER, VARCHAR, DATE, BOOLEAN, JSON) which dictates the kind of data it can store and how it is stored. Data types can vary slightly between different database systems.
  • Primary Keys (PK): A column or combination of columns that uniquely identifies each row in a table. Ensures entity integrity.
  • Foreign Keys (FK): A column or combination of columns in one table that refers to the primary key in another table, establishing and enforcing a link or relationship between the tables. Ensures referential integrity.
  • Constraints: Rules defined on columns or tables to enforce data integrity and business rules. Common constraints include: NOT NULL, UNIQUE, CHECK, PRIMARY KEY, FOREIGN KEY.
  • Normalization: The process of organizing database tables to reduce data redundancy and improve data integrity.
  • Relationships: Connections between tables, typically categorized as One-to-One, One-to-Many, or Many-to-Many.
  • More on schema: A schema includes:
    • Table definitions (names, columns, data types)
    • Relationships between tables
    • Constraints and rules
    • Indexes for performance optimization
    • Views and stored procedures
    • Security and access permissions
    A schema serves as a logical container for database objects and provides a way to organize and manage database elements. It defines the structure of the database, including what tables exist, what columns each table has, and how tables relate to each other. Schemas help in:
    • Organizing database objects into logical groups
    • Managing access permissions
    • Maintaining data integrity
    • Supporting multiple users and applications
    • Facilitating database maintenance and updates

SQL History

SQL (Structured Query Language) has a rich history that dates back to the 1970s:

1970s - The Birth of SQL

  • Developed by Donald D. Chamberlin and Raymond F. Boyce at IBM in the early 1970s
  • Originally called SEQUEL (Structured English Query Language)
  • First implemented in IBM's System R project
  • Designed to manipulate and retrieve data stored in IBM's System R database

1980s - Standardization and Growth

  • 1986: First SQL standard published by ANSI (SQL-86 or SQL-87)
  • 1989: First revision of the standard (SQL-89)
  • IBM releases DB2, the first commercial SQL database
  • Oracle, Sybase, and other vendors begin implementing SQL in their products

1990s - Maturity and Expansion

  • 1992: Major revision of the standard (SQL-92 or SQL2)
  • Introduction of new features like joins, subqueries, and transaction management
  • 1999: SQL:1999 standard introduces object-oriented features and recursive queries
  • Major database vendors implement SQL-92 features

2000s - Modern Developments

  • 2003: SQL:2003 standard adds XML support and window functions
  • 2006: SQL:2006 standard enhances XML capabilities
  • 2008: SQL:2008 standard introduces TRUNCATE TABLE and INSTEAD OF triggers
  • 2011: SQL:2011 standard adds temporal database features
  • 2016: SQL:2016 standard introduces JSON support and row pattern matching

Key Milestones

  • SQL became an ISO standard in 1987
  • The language has evolved while maintaining backward compatibility
  • Different database vendors have implemented various SQL dialects
  • SQL remains the most widely used database language today

Impact and Legacy

  • SQL has become the standard language for relational database management systems
  • Used by millions of developers worldwide
  • Forms the foundation for modern database systems
  • Continues to evolve to meet new data management challenges

SQL (Structured Query Language)

SQL is the standard language used to communicate with relational databases. It allows users to create, retrieve, update, and delete data (CRUD operations), as well as manage the database structure itself. SQL commands are broadly categorized:

  • DDL (Data Definition Language): Used to define or modify the database structure (e.g., CREATE TABLE, ALTER TABLE, DROP TABLE).
  • DML (Data Manipulation Language): Used to query and modify data (e.g., SELECT, INSERT, UPDATE, DELETE).
  • DCL (Data Control Language): Used to manage permissions and access control (e.g., GRANT, REVOKE).

The Broader RDBMS Landscape

While SQL is the standard language, different vendors implement RDBMS with variations in features, performance characteristics, licensing, and minor SQL syntax differences (dialects). It is important to be aware of the major players in the RDBMS market.

Key takeaways from the comparison:

  • PostgreSQL: Known for its standards compliance, extensibility, and rich feature set.
  • MySQL: Widely popular, especially for web applications.
  • Oracle Database: Market leader in the enterprise space, known for robustness.
  • Microsoft SQL Server: Strong contender, particularly in Windows environments.
  • IBM Db2: Long-standing enterprise database, often in large corporations.

Understanding these differences helps in choosing the right database and adapting SQL skills.

Role of Databases in ERP Systems

Enterprise Resource Planning (ERP) systems like Odoo rely heavily on robust database systems.

  • Central Data Repository: Consolidates data from various departments.
  • Data Integrity and Consistency: Enforces business rules and constraints.
  • Process Automation and Workflow: Drives automated workflows based on data.
  • Reporting and Business Intelligence: Enables powerful querying and analytics.
  • Security and Access Control: Manages user permissions.
  • Scalability and Performance: Handles large data volumes and transaction rates.

Specific Example: Odoo and PostgreSQL: Odoo uses PostgreSQL, leveraging features like schemas, rich data types (JSONB), transactions, and extensibility. The Odoo schema is complex, representing various interconnected business objects.

Understanding the database structure is crucial for product owners to:

  • Understand data dependencies.
  • Define requirements accurately.
  • Specify reporting needs effectively.
  • Communicate clearly with technical teams.
  • Appreciate performance considerations.
