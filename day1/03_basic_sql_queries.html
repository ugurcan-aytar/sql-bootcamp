Day 1: Basic SQL Queries

Basic SQL Queries

SELECT Statements and Basic Syntax

The SELECT statement is the cornerstone of SQL, used to retrieve data from database tables. Mastering its syntax is essential for querying any relational database.

Basic SELECT Syntax (Standard SQL):

SELECT column1, column2, ...
FROM table_name;

This retrieves specific columns from the specified table.

Selecting All Columns (SELECT *):

SELECT * FROM table_name;

While convenient for quick exploration (SELECT * is standard SQL), using it in application code is generally discouraged across all database systems because it: - Can retrieve more data than needed, impacting network traffic and performance. - Makes code less readable as it obscures which columns are actually used. - Can break applications if the table structure changes (e.g., columns added/removed).

Selecting Specific Columns: Always prefer listing the specific columns you need:

-- Example using Odoo schema (PostgreSQL)
SELECT name, email, phone
FROM res_partner;

Column Aliases (Standard SQL): Aliases provide temporary, often more readable, names for columns or expressions in the query result. The AS keyword is standard but often optional.

SELECT 
    name AS "Customer Name", -- Quoted alias preserves case/spaces (standard)
    email AS EmailAddress,   -- Unquoted alias (case usually ignored/lowercased)
    phone MobilePhone        -- AS keyword is optional
FROM res_partner;
  • Dialect Note: While double quotes for aliases are standard SQL, some systems use different quoting (e.g., backticks ` in MySQL, square brackets [] in SQL Server). Unquoted aliases are generally case-insensitive.

Expressions in SELECT (Standard SQL): You can perform calculations or use functions on columns directly within the SELECT list.

-- Example using Odoo schema (PostgreSQL)
SELECT 
    product_id,
    price_unit,
    product_uom_qty, -- Odoo often uses product_uom_qty for quantity
    price_unit * product_uom_qty AS line_total
FROM sale_order_line;

DISTINCT Keyword (Standard SQL): DISTINCT eliminates duplicate rows from the result set. It considers the entire row based on the selected columns.

-- Get unique country IDs from partners
SELECT DISTINCT country_id
FROM res_partner;

DISTINCT ON (PostgreSQL Extension): PostgreSQL offers DISTINCT ON (expression) which returns the first row for each unique value of the expression, based on the ORDER BY clause. This is very useful but not standard SQL.

-- Get one partner per country (the most recently created one)
-- PostgreSQL specific syntax!
SELECT DISTINCT ON (country_id) country_id, id, name
FROM res_partner
ORDER BY country_id, create_date DESC;
  • Standard Alternative: Achieving the same result in standard SQL usually involves window functions like ROW_NUMBER(), which we will cover later.

Filtering with WHERE Clauses (Standard SQL)

The WHERE clause filters rows before any grouping or aggregation happens, based on specified conditions.

Basic Syntax:

SELECT column_list
FROM table_name
WHERE condition;

Comparison Operators (Standard SQL): - Equal: = - Not equal: <> (standard), != (common extension) - Greater than: > - Less than: < - Greater than or equal to: >= - Less than or equal to: <=

Examples:

-- Partners from the United States (assuming country_id 233 is US)
SELECT name, email, city
FROM res_partner
WHERE country_id = 233;

-- Products with a list price greater than 100
SELECT name, list_price
FROM product_template
WHERE list_price > 100;

-- Orders placed on or after a specific date
SELECT name, date_order
FROM sale_order
WHERE date_order >= '2023-01-01'; -- Date format 'YYYY-MM-DD' is standard

Logical Operators (Standard SQL): - AND: Both conditions must be true. - OR: At least one condition must be true. - NOT: Reverses the truth value of a condition. Use parentheses () to control the order of evaluation for complex conditions.

Examples:

-- Partners from the US OR Canada (IDs 233, 20)
SELECT name, country_id
FROM res_partner
WHERE country_id = 233 OR country_id = 20;

-- Partners from the US OR Canada (IDs 233, 20)
SELECT name, country_id
FROM res_partner
WHERE country_id IN (233, 20);

-- Active products that are in stock
SELECT name, sq.quantity
FROM product_template pt
LEFT JOIN product_product pp on pt.id = pp.product_tmpl_id
LEFT JOIN stock_quant sq on pp.id = sq.product_id
WHERE pt.active = true AND sq.quantity > 0;

-- Orders that are NOT cancelled
SELECT name, state
FROM sale_order
WHERE NOT state = 'cancel';
-- Equivalent using <> or !=
-- WHERE state <> 'cancel';

Handling NULL Values (Standard SQL): NULL represents an unknown or missing value. It cannot be compared using standard operators like = or <>. Use IS NULL or IS NOT NULL instead. 

-- Partners missing an email address
SELECT name
FROM res_partner
WHERE email IS NULL;

-- Partners who have an email address
SELECT name, email
FROM res_partner
WHERE email IS NOT NULL;
  • Important: WHERE email = NULL or WHERE email <> NULL will not work as expected in standard SQL because NULL is not equal (or unequal) to anything, including itself.

BETWEEN Operator (Standard SQL): Checks if a value falls within a specified range (inclusive). 

SELECT name, list_price
FROM product_template
WHERE list_price BETWEEN 100 AND 500; -- Equivalent to: list_price >= 100 AND list_price <= 500

IN Operator (Standard SQL): Checks if a value matches any value in a provided list.

SELECT name, country_id
FROM res_partner
WHERE country_id IN (235, 38, 73); -- US, Canada, Germany

This is often more readable than multiple OR conditions.

LIKE Operator (Standard SQL): Performs simple pattern matching on string data. - %: Matches any sequence of zero or more characters. - _: Matches exactly one character. 

-- Partner names starting with 'Geo'
SELECT name
FROM res_partner
WHERE name LIKE 'Geo%';

-- Emails ending with '.org'
SELECT name, email
FROM res_partner
WHERE email LIKE '%.org';

-- Product codes with 'X' as the second character (e.g., 'AX123')
SELECT default_code
FROM product_template
WHERE default_code LIKE '_-%';
  • Case Sensitivity: Standard SQL LIKE is case-sensitive. However, the default behavior varies across databases and depends on collation settings (e.g., often case-insensitive by default in SQL Server and MySQL, case-sensitive in PostgreSQL and Oracle).
  • Case-Insensitive LIKE:
    • PostgreSQL: Use ILIKE (e.g., WHERE name ILIKE 'geo%').
    • Other DBs: Often achieved using functions like LOWER() or UPPER() on both sides (e.g., WHERE LOWER(name) LIKE 'geo%') or by setting case-insensitive collations.

Regular Expressions (Not Standard SQL, Syntax Varies): Most databases support more powerful pattern matching using regular expressions, but the syntax varies significantly.

-- PostgreSQL Example: Names containing numbers
SELECT name, email FROM res_partner WHERE email ~ '[0-9]';

-- PostgreSQL Example: Case-insensitive regex for names starting with 'a'
SELECT name FROM res_partner WHERE name ~* '^a'; 

-- MySQL Example: Using REGEXP or RLIKE
-- SELECT name FROM res_partner WHERE name REGEXP '[0-9]';

-- Oracle Example: Using REGEXP_LIKE
-- SELECT name FROM res_partner WHERE REGEXP_LIKE(name, '[0-9]');

-- SQL Server Example: Using PATINDEX (limited regex)
-- SELECT name FROM res_partner WHERE PATINDEX('%[0-9]%', name) > 0;

Sorting with ORDER BY (Standard SQL)

The ORDER BY clause sorts the rows in the result set.

Basic Syntax:

SELECT column_list
FROM table_name
WHERE condition -- Optional
ORDER BY column1 [ASC|DESC], column2 [ASC|DESC], ...;

Sort Direction: - ASC: Ascending order (A-Z, 0-9). This is the default if not specified. - DESC: Descending order (Z-A, 9-0).

Examples:

-- Sort partners alphabetically by name
SELECT name, city
FROM res_partner
ORDER BY name ASC; -- ASC is optional here

-- Sort products by price, highest first
SELECT name, list_price
FROM product_template
ORDER BY list_price DESC;

-- Sort partners by country, then by city within each country
SELECT name, country_id, city
FROM res_partner
ORDER BY country_id ASC, city ASC;

-- Sort by an expression (e.g., length of name)
SELECT name
FROM res_partner
ORDER BY LENGTH(name) DESC; -- Function name might vary (e.g., LEN in SQL Server)

Handling NULLs in Sorting (Behavior Varies): How NULL values are sorted (FIRST or LAST) depends on the database system when ASC or DESC is used. - PostgreSQL/Oracle: Default is NULLS LAST for ASC, NULLS FIRST for DESC. - SQL Server/MySQL: Default is NULLS FIRST for ASC, NULLS LAST for DESC. - Explicit Control (PostgreSQL/Oracle): You can explicitly specify NULLS FIRST or NULLS LAST. sql -- PostgreSQL/Oracle syntax SELECT name, email FROM res_partner ORDER BY email ASC NULLS FIRST; SELECT name, email FROM res_partner ORDER BY email DESC NULLS LAST;

Limiting Results (Syntax Varies)

Restricting the number of rows returned is common, especially for pagination. The syntax has historically varied but is converging towards the SQL standard.

PostgreSQL / MySQL Syntax:

SELECT column_list
FROM table_name
ORDER BY sort_column -- ORDER BY is crucial for predictable LIMIT
LIMIT row_count [OFFSET starting_row];
  • LIMIT row_count: Specifies the maximum number of rows to return.
  • OFFSET starting_row: Specifies the number of rows to skip before starting to return rows (0-based).

Examples (PostgreSQL/MySQL):

-- Get the top 10 most recently created partners
SELECT name, create_date
FROM res_partner
ORDER BY create_date DESC
LIMIT 10;

-- Get the third page of 25 partners (rows 51-75)
SELECT name
FROM res_partner
ORDER BY name
LIMIT 25 OFFSET 25; -- Skip 25 rows (1 page * 25 rows/page)

SQL Standard Syntax (PostgreSQL 8.4+, Oracle 12c+, SQL Server 2012+, Db2):

SELECT column_list
FROM table_name
ORDER BY sort_column
OFFSET starting_row ROWS
FETCH FIRST row_count ROWS ONLY; -- or FETCH NEXT ...

Example (Standard SQL):

-- Get the top 10 most recently created partners
SELECT name, create_date
FROM res_partner
ORDER BY create_date DESC
FETCH FIRST 10 ROWS ONLY;

-- Get the second page of 25 partners (rows 26-50)
SELECT name
FROM res_partner
ORDER BY name
OFFSET 25 ROWS FETCH NEXT 25 ROWS ONLY;

Older Vendor-Specific Syntax: - SQL Server (Older): SELECT TOP row_count column_list ... ORDER BY ... (No easy OFFSET) - Oracle (Older): SELECT column_list ... WHERE ROWNUM <= row_count ... ORDER BY ... (Complex for OFFSET)

Practical Examples (Using Odoo/PostgreSQL Schema)

These examples use PostgreSQL syntax but illustrate common data retrieval tasks applicable to many business databases.

Example 1: Find recent, high-value sales orders

SELECT 
    so.name AS order_ref,
    rp.name AS customer_name,
    so.date_order,
    so.amount_total
FROM 
    sale_order so
JOIN 
    res_partner rp ON so.partner_id = rp.id
WHERE 
    so.state IN ('sale', 'done') -- Consider only confirmed/done orders
    AND so.date_order >= CURRENT_DATE - INTERVAL '90 days' -- Orders in the last 90 days
    AND so.amount_total > 1000 -- Orders over $1000
ORDER BY 
    so.amount_total DESC
LIMIT 20;
  • Note: CURRENT_DATE and INTERVAL syntax might differ slightly in other databases.

Example 2: Find products low on stock

SELECT 
    pt.name AS product_name,
    pt.default_code AS internal_reference,
    SUM(sq.quantity) AS quantity_on_hand -- Assumes stock.quant stores current levels
FROM 
    product_template pt
JOIN 
    product_product pp ON pp.product_tmpl_id = pt.id
LEFT JOIN -- Use LEFT JOIN in case a product has no stock record yet
    stock_quant sq ON sq.product_id = pp.id -- Need to consider location too in real scenario
WHERE 
    pt.active = true
    AND pt.type = 'consu' -- Filter for storable products
GROUP BY -- Need GROUP BY when using SUM()
    pt.id, pt.name, pt.default_code
HAVING -- Filter *after* aggregation
    SUM(sq.quantity) < 10 -- Assuming reorder point is 10
ORDER BY 
    quantity_on_hand ASC;
  • Note: GROUP BY and HAVING will be covered in more detail later. The concept of querying inventory levels is common across systems.

Example 3: Find customers with specific email domains

SELECT 
    name,
    email
FROM 
    res_partner
WHERE 
    email IS NOT NULL 
    AND (
        email LIKE '%@example.com' 
        OR email LIKE '%@sample.org'
    )
ORDER BY 
    name
LIMIT 50;
  • Note: This uses standard LIKE and should work across most databases, assuming case-sensitivity matches requirements or appropriate functions/collations are used.

Understanding these basic SELECT statement components (SELECT, FROM, WHERE, ORDER BY, LIMIT/FETCH) is the first major step in learning SQL. They allow you to retrieve, filter, and sort data from any relational database.
