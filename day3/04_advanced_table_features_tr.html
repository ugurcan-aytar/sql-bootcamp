Bölüm 3: Gelismis Tablo Ozellikleri

Modern SQL Tablo Yetenekleri

Bu oturum, daha verimli ve bakimi kolay veritabanlari tasarlamaniza yardimci olan PostgreSQL'deki gelismis tablo ozelliklerini kapsar. Bu ozellikler hesaplamalari otomatiklestirir, performansi arttirir ve veri yapilariniz uzerinde daha iyi kontrol saglar.

Olusturulan Sutunlar (Hesaplanan Sutunlar)

Olusturulan sutun, degeri ayni satirdaki diger sutunlardan otomatik olarak hesaplanan bir sutundur. Uygulamanizda veya sorgularinizda deger hesaplamak yerine, veritabani bunlari otomatik olarak korur.

Temel Faydalar:
  • Sorgularda gereksiz hesaplamalari ortadan kaldirir
  • Veri tutarliligini saglar (hesaplanan deger her zaman kaynak sutunlarla eslesir)
  • Hizli aramalar icin indekslenebilir
  • Uygulama karmasikligini azaltir

Sozdizimi

CREATE TABLE tablo_adi (
    sutun1 veri_turu,
    sutun2 veri_turu,
    hesaplanan_sutun veri_turu GENERATED ALWAYS AS (ifade) STORED
);

STORED, degerin satir eklendiginde veya guncellendiginde hesaplandigi ve diskte saklananigi anlamina gelir. PostgreSQL yalnizca STORED olusturulan sutunlari destekler.

Ornekler

Ornek 1: Ad ve Soyaddan Tam Ad

CREATE TABLE kisiler (
    id SERIAL PRIMARY KEY,
    ad VARCHAR(50) NOT NULL,
    soyad VARCHAR(50) NOT NULL,
    tam_ad VARCHAR(101) GENERATED ALWAYS AS (ad || ' ' || soyad) STORED,
    email VARCHAR(255)
);

INSERT INTO kisiler (ad, soyad, email)
VALUES ('Ahmet', 'Yilmaz', 'ahmet@ornek.com');

SELECT * FROM kisiler;
-- tam_ad otomatik olarak 'Ahmet Yilmaz'

Ornek 2: Siparis Satiri Toplam Fiyati

-- bootcamp_db'deki order_lines'a benzer
CREATE TABLE siparis_kalemleri (
    id SERIAL PRIMARY KEY,
    siparis_id INTEGER NOT NULL,
    urun_id INTEGER NOT NULL,
    miktar INTEGER NOT NULL CHECK (miktar > 0),
    birim_fiyat DECIMAL(10,2) NOT NULL,
    indirim_yuzdesi DECIMAL(5,2) DEFAULT 0,
    -- Otomatik hesaplanan satir toplami
    satir_toplami DECIMAL(12,2) GENERATED ALWAYS AS (
        miktar * birim_fiyat * (1 - indirim_yuzdesi / 100)
    ) STORED
);

INSERT INTO siparis_kalemleri (siparis_id, urun_id, miktar, birim_fiyat, indirim_yuzdesi)
VALUES (1, 101, 3, 99.99, 10);

SELECT miktar, birim_fiyat, indirim_yuzdesi, satir_toplami FROM siparis_kalemleri;
-- satir_toplami = 3 * 99.99 * 0.90 = 269.97
Olusturulan Sutun Sinirliliklari:
  • Diger olusturulan sutunlara basvuramaz
  • Diger tablolardaki sutunlara basvuramaz
  • Alt sorgular veya volatile fonksiyonlar (RANDOM() gibi) kullanilamaz
  • Dogrudan guncellenemez - kaynak sutunlar guncellenmeli

Olusturulan Sutunlari Indeksleme

Hizli aramalar icin olusturulan sutunlar uzerinde indeksler olusturabilirsiniz:

-- Hizli ad aramalari icin tam_ad uzerinde indeks
CREATE INDEX idx_kisiler_tam_ad ON kisiler(tam_ad);

-- Artik bu sorgu indeksi kullaniyor
SELECT * FROM kisiler WHERE tam_ad = 'Ahmet Yilmaz';

Kimlik Sutunlari (Modern Otomatik Artis)

Kimlik sutunlari, otomatik artan sutunlar olusturmanin SQL standart yoludur. Eski SERIAL turundan daha fazla kontrol saglar ve yeni tasarimlar icin onerilir.

Kimlik vs SERIAL Karsilastirmasi

Ozellik SERIAL IDENTITY
SQL Standardi PostgreSQL'e ozgu SQL:2003 standardi
Sekans sahipligi Ayri sekans nesnesi Sutuna bagli
DROP davranisi Sekans ayri silinmeli Sekans sutunla silinir
Ekleme geersiz kilma Her zaman izin verilir Kontrol edilir (ALWAYS vs BY DEFAULT)
Tasinabilirlik Sadece PostgreSQL SQL Server, Oracle vb. ile calisir

Kimlik Sutunu Sozdizimi

-- GENERATED ALWAYS: Deger gecersiz kilinamaz (kati)
CREATE TABLE urunler_yeni (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    ad VARCHAR(100) NOT NULL,
    fiyat DECIMAL(10,2)
);

-- GENERATED BY DEFAULT: Gerekirse gecersiz kilinabilir (esnek)
CREATE TABLE siparisler_yeni (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    musteri_id INTEGER NOT NULL,
    siparis_tarihi DATE DEFAULT CURRENT_DATE
);
En Iyi Uygulama: Manuel ID atamasi istemediginiz birincil anahtarlar icin GENERATED ALWAYS AS IDENTITY kullanin. Veri tasimalari veya ozel durumlar icin esneklik gerektiginde GENERATED BY DEFAULT kullanin.

Gecici Tablolar

Gecici tablolar yalnizca oturumunuz veya isleminiz suresince vardir. Ara sonuclari saklamak, veri hazirligi veya karmasik hesaplamalar icin kullnislidir.

Gecici Tablo Olusturma

-- Temel gecici tablo (oturum icin vardir)
CREATE TEMPORARY TABLE gecici_hesaplamalar (
    id SERIAL PRIMARY KEY,
    musteri_id INTEGER,
    toplam_tutar DECIMAL(12,2),
    hesaplama_tarihi DATE DEFAULT CURRENT_DATE
);

-- Kisa form
CREATE TEMP TABLE gecici_sonuclar (
    urun_id INTEGER,
    satilan_miktar INTEGER
);

Tablo Yasam Suresini Kontrol Etme

ON COMMIT kullanarak gecici tablo verilerinin ne zaman temizlenecegini kontrol edebilirsiniz:

-- Her islem sonunda veriler silinir (satirlar temizlenir, tablo kalir)
CREATE TEMP TABLE gecici_yigin_verisi (
    oege_id INTEGER,
    durum VARCHAR(20)
) ON COMMIT DELETE ROWS;

-- Islem sonunda tablo tamamen silinir
CREATE TEMP TABLE sadece_islem_icin (
    hesap_id INTEGER,
    sonuc DECIMAL(10,2)
) ON COMMIT DROP;

-- Varsayilan: Oturum sonuna kadar veriler korunur (PRESERVE ROWS)
CREATE TEMP TABLE gecici_oturum_verisi (
    veri_id INTEGER,
    deger TEXT
) ON COMMIT PRESERVE ROWS;
Gecici Tablo Ozellikleri:
  • Yalnizca oturumunuz tarafindan gorulebilir (diger kullanicilar goremez)
  • Kalici tablolarla ayni ada sahip olabilir (gecici oncelik alir)
  • Indeksler, kisitlamalar ve tetikleyiciler iceribilir
  • Varsayilan olarak WAL'a yazilmaz (daha hizli yazmalar)
  • Oturum sonlandiginda otomatik olarak silinir

Loglanmayan Tablolar

Loglanmayan tablolar, Write-Ahead Log'a (WAL) yazmayan kalici tablolardir. Bu, yazma islemlerini onemli olcude hizlandirir, ancak sunucu cokerse veriler kaybolur.

Loglanmayan Tablo Olusturma

CREATE UNLOGGED TABLE oturum_onbellek (
    oturum_id VARCHAR(64) PRIMARY KEY,
    kullanici_id INTEGER,
    veri JSONB,
    son_kullanim TIMESTAMP
);

CREATE UNLOGGED TABLE iceri_aktarim_hazirlama (
    satir_no SERIAL,
    ham_veri TEXT,
    islendi BOOLEAN DEFAULT FALSE,
    hata_mesaji TEXT
);

Loglanmayan Tablolari Ne Zaman Kullanmali

Iyi Kullanim Alanlari Kotu Kullanim Alanlari
Oturum/onbellek verileri Finansal islemler
Iceri aktarim hazirlama alanlari Siparis kayitlari
Gecici hesaplama sonuclari Musteri verileri
Kuyruk tablolari (kayip kabul edilebilirse) Denetim kayitlari
Test/gelistirme verileri Kritik is verileri
Kritik Uyari: Loglanmayan tablolar, veritabani cokmesi veya temiz olmayan kapatma sonrasinda kesilerek bosaltilir. Kaybetmeyi goze alamayacaginiz veriler icin asla kullanmayin!

Tablo Klonlama (CREATE TABLE ... LIKE)

CREATE TABLE ... LIKE ifadesi, mevcut bir tabloyla ayni yapida yeni bir tablo olusturur. Yedek tablolar, hazirlama tablolari veya bolumler olusturmak icin kullanislidir.

Temel Sozdizimi

-- Ayni sutun ve veri turleriyle tablo olustur
CREATE TABLE musteriler_yedek (LIKE customers);

-- Bu, ayni sutun tanimlarına sahip bos bir tablo olusturur
-- ancak varsayilan olarak veri, indeks veya kisitlama YOKTUR

INCLUDING Secenekleri

Kaynak tablodan cesitli ozellikleri dahil edebilirsiniz:

-- Belirli ozellikleri dahil et
CREATE TABLE siparisler_arsiv (LIKE orders INCLUDING DEFAULTS);
CREATE TABLE urunler_kopya (LIKE products INCLUDING INDEXES);
CREATE TABLE musteriler_klon (LIKE customers INCLUDING CONSTRAINTS);

-- Her seyi dahil et
CREATE TABLE tam_yedek (LIKE orders INCLUDING ALL);

-- Birden fazla secenek
CREATE TABLE hazirlama_siparisler (
    LIKE orders
    INCLUDING DEFAULTS
    INCLUDING CONSTRAINTS
    EXCLUDING INDEXES  -- Indeksleri kopyalama (daha hizli toplu ekleme)
);

Ertelenebilir Kisitlamalar

Ertelenebilir kisitlamalar, kisitlama kontrolunu islem sonuna kadar ertelemenize olanak tanir. Bu, bir dizi islem sirasinda kisitlamalari gecici olarak ihlal etmeniz gerektiginde kullanislidir.

Problem

Iki calisanin ID'lerini degistirmeniz gereken bir senaryo dusunun:

-- Normal UNIQUE kisitlamayla bu basarisiz olur!
-- Calisan 1'in ID'si 100, Calisan 2'nin ID'si 200
-- Bunlari degistirmek istiyoruz

UPDATE calisanlar SET id = 200 WHERE id = 100;  -- BASARISIZ! 200 zaten var
UPDATE calisanlar SET id = 100 WHERE id = 200;  -- Yine basarisiz olurdu

Ertelenebilir Kisitlama Olusturma

CREATE TABLE departmanlar_v2 (
    id INTEGER PRIMARY KEY,
    ad VARCHAR(100) NOT NULL,
    yonetici_id INTEGER,
    -- Ertelenebilir benzersiz kisitlama
    CONSTRAINT uk_dept_yonetici UNIQUE (yonetici_id)
        DEFERRABLE INITIALLY DEFERRED
);

Kisitlama Modlari

Mod Ne Zaman Kontrol Edilir Kullanim Alani
NOT DEFERRABLE Hemen (varsayilan) Normal kisitlamalar
DEFERRABLE INITIALLY IMMEDIATE Hemen, ama ertelenebilir Esnek - varsayilan hemen
DEFERRABLE INITIALLY DEFERRED Islem onayinda Karmasik cok adimli islemler

Ertelenebilir Kisitlamalari Kullanma

-- Mevcut islem icin kisitlama modunu ayarla
BEGIN;

-- Tum ertelenebilir kisitlamalari ertele
SET CONSTRAINTS ALL DEFERRED;

-- Artik kisitlamayi gecici olarak ihlal edebiliriz
UPDATE departmanlar_v2 SET yonetici_id = 999 WHERE id = 1;  -- gecici deger
UPDATE departmanlar_v2 SET yonetici_id = 1 WHERE id = 2;    -- dept 1'in yoneticisiydi
UPDATE departmanlar_v2 SET yonetici_id = 2 WHERE id = 1;    -- degisim tamamlandi

-- Kisitlama burada kontrol edilir - simdi gecerli olmali
COMMIT;
Ertelenebilir Kisitlamalari Ne Zaman Kullanmali:
  • Satirlar arasinda benzersiz degerleri degistirme
  • Dairesel basvurularla veri ekleme
  • Karmasik veri tasimalari
  • Kisitlamalari gecici olarak ihlal eden toplu islemler
