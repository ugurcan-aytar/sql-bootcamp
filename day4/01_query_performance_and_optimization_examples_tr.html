Sorgu Performansi ve Optimizasyon - Pratik Ornekler

Uygulamali alistirmalarla yurutme planlarini, indeksleme stratejilerini ve sorgu optimizasyon tekniklerini ogrenmek.

Ogrenme Hedefleri:
  • EXPLAIN ciktisini okumak ve yorumlamak
  • Yurutme planlarindaki performans darbogazlarini tanimlamak
  • Farkli sorgu kaliplari icin etkili indeksler tasarlamak
  • Yavas sorgulari daha iyi performans icin yeniden yazmak

Ornek Veritabani Semasi

Asagidaki tablolara sahip bir e-ticaret veritabani kullanacagiz:

customers
Sutun Tip Notlar
customer_id SERIAL PK Birincil Anahtar
email VARCHAR(100) Benzersiz
first_name VARCHAR(50)
last_name VARCHAR(50)
country VARCHAR(50) ~50 farkli deger
is_active BOOLEAN %90 TRUE
created_at TIMESTAMP

~500.000 satir

orders
Sutun Tip Notlar

customer_id INTEGER FK Yabanci Anahtar
order_date DATE 2020-2024
status VARCHAR(20) 5 deger
total_amount DECIMAL(10,2)
shipping_country VARCHAR(50)

~2.000.000 satir

Sutun Tip Notlar

product_id INTEGER FK Yabanci Anahtar

Veritabani yalnizca birincil anahtar indeksleriyle baslar. Yabanci anahtar sutunlarinda henuz indeks yoktur.

Bolum 1: Yurutme Planlarini Okuma

EXPLAIN ciktisini yorumlamayi ve performans sorunlarini tanimlama yi ogrenin.

Alistirma 1.1: Temel Plan Islemlerini Anlama

Asagidaki PostgreSQL EXPLAIN ciktisina gore, islemleri tanimlayin ve performans ozelliklerini tahmin edin:

EXPLAIN SELECT * FROM orders WHERE status = 'pending';

Seq Scan on orders  (cost=0.00..45123.00 rows=150000 width=52)
  Filter: ((status)::text = 'pending'::text)

Sorular:

  1. Hangi islem gerceklestiriliyor?
  2. 2 milyon satirlik bir tablo icin neden verimsiz olabilir?
  3. Maliyet tahmini (0.00..45123.00) neyi temsil ediyor?

Cevaplar:

  1. Seq Scan - Bir Sequential Scan (tam tablo taramasi) tablodaki her satiri okur.
  2. Sorgu yalnizca ~150.000 satir (%7.5 / 2M) gerektiriyor, ancak TUM 2 milyon satiri okuyor. Bu, I/O ve CPU israfina neden olur.
  3. Maliyet iki sayiya sahiptir:
    • 0.00 = baslangic maliyeti (ilk satir donmeden onceki is)
    • 45123.00 = keyfi birimlerde toplam maliyet (yuksek = yavas)

Optimizasyon: status sutununda bir indeks olusturun:

CREATE INDEX idx_orders_status ON orders (status);

Alistirma 1.2: Plan Islemlerini Karsilastirma

Indeks eklendikten sonra ayni sorgu icin bu iki yurutme planini karsilastirin:

Once (Indeks Yok):
Seq Scan on orders
  (cost=0.00..45123.00 rows=150000)
  Filter: status = 'pending'

Actual Time: 892.45ms
Rows Returned: 148,532
Sonra (Indeks Var):
Bitmap Heap Scan on orders
  (cost=1523.00..28456.00 rows=150000)
  Recheck Cond: status = 'pending'
  -> Bitmap Index Scan on idx_orders_status
       (cost=0.00..1485.00 rows=150000)

Actual Time: 124.32ms
Rows Returned: 148,532

Sorular:

  1. Optimize edilmis planda hangi yeni islemler gorunuyor?
  2. Performans iyilestirmesini hesaplayin.
  3. PostgreSQL neden basit bir Index Scan yerine Bitmap scan kullaniyor?

Cevaplar:

  1. Bitmap Index Scan - Eslesen satirlarin bir bitmap'ini olusturmak icin indeksi tarar
    Bitmap Heap Scan - Tablodan gercek satirlari almak icin bitmap'i kullanir
  2. Performans iyilestirmesi: 892.45ms -> 124.32ms = 7.2x daha hizli
    Maliyet azalmasi: 45123 -> 28456 = %37 daha dusuk tahmini maliyet
  3. PostgreSQL, cok sayida satir alirken (%7.5 tablo) Bitmap Scan kullanir. Buyuk sonuc kumeleri icin Index Scan'den daha verimlidir cunku:
    • Satir konumlarini siralayarak rastgele I/O'yu azaltir
    • Birden fazla indeks kosulunu verimli bir sekilde birlestirebilir
    • Basit bir Index Scan cok fazla rastgele disk aramasina neden olur

Alistirma 1.3: Join Islemlerini Tanimlama

Bir join sorgusu icin bu yurutme planini analiz edin:

SELECT c.email, o.order_date, o.total_amount
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
WHERE c.country = 'Germany'
  AND o.order_date >= '2024-01-01';
Hash Join  (cost=15234.00..98765.00 rows=25000 width=48)
  Hash Cond: (o.customer_id = c.customer_id)
  ->  Seq Scan on orders o  (cost=0.00..45123.00 rows=500000 width=20)
        Filter: (order_date >= '2024-01-01')
  ->  Hash  (cost=12456.00..12456.00 rows=12500 width=36)
        ->  Seq Scan on customers c  (cost=0.00..12456.00 rows=12500 width=36)
              Filter: (country = 'Germany')

Sorular:

  1. Hangi join algoritmasi kullaniliyor?
  2. Tum sequential scan'leri tanimlayin - hangileri indekslerden fayda saglayabilir?
  3. Bu sorguyu optimize etmek icin hangi indeksleri olusturursunuz?

Cevaplar:

  1. Hash Join - Daha kucuk sonuctan (Almanya'daki musteriler) bir hash tablosu olusturur ve bunu daha buyuk sonucla (siparisler) sorgular.
  2. Iki sequential scan tanimlandi:
    • customers - country = 'Germany' ile filtrelenmis - Indeksten faydalanir
    • orders - order_date >= '2024-01-01' ile filtrelenmis - Indeksten faydalanir
  3. Onerilen indeksler: 
    -- Musteri ulke filtresi icin indeks
CREATE INDEX idx_customers_country ON customers (country);

-- Siparis tarih araligi filtresi icin indeks
CREATE INDEX idx_orders_date ON orders (order_date);

-- Join kosulu icin indeks (kritik!)
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

Indekslerden sonra, plan cok daha iyi performans icin Index Scan'lerle Nested Loop kullanabilir.

Bolum 2: Indeks Tasarimi Alistirmalari

Farkli sorgu kaliplari icin optimal indeksler tasarlamayi pratik yapin.

Alistirma 2.1: Tek Sutun vs Bilesik Indeks

Sik calistirilan su iki sorgunuz var:

-- Sorgu A: Duruma gore siparisleri bul
SELECT * FROM orders WHERE status = 'shipped';

-- Sorgu B: Tarih araliginda gonderilmis siparisleri bul
SELECT * FROM orders
WHERE status = 'shipped'
  AND order_date BETWEEN '2024-01-01' AND '2024-03-31';

Gorev: Her iki sorguyu da verimli bir sekilde desteklemek icin optimal indeks(ler)i tasarlayin.

Ipucu: Bilesik indekslerdeki sutun sirasini dikkate alin. Esitlik kosullari aralik kosullarindan once gelmelidir.

Optimal Cozum: BIR bilesik indeks olusturun:

CREATE INDEX idx_orders_status_date ON orders (status, order_date);

Neden calisiyor:

  • Sorgu A: En soldaki sutunu (status) kullanir - indeks kullanilir
  • Sorgu B: Her iki sutunu da kullanir - once status esitligi, sonra tarih araligi

Neden ayri indeksler degil?

-- Daha az optimal yaklasim:
CREATE INDEX idx_orders_status ON orders (status);
CREATE INDEX idx_orders_date ON orders (order_date);
  • Sorgu B'nin iki indeksi birlestirmesi (Bitmap AND) veya yalnizca birini secmesi gerekir
  • Daha fazla indeks = daha fazla depolama ve daha yavas yazmalar
Sutun Sirasi Onemlidir!
(order_date, status) uzerindeki bir indeks, Sorgu A'yi verimli bir sekilde desteklemez cunku en soldaki sutun kullanilmaz.

Alistirma 2.2: Kismi Indeks Tasarimi

Uygulamaniz sik sik su sorguyu calistiriyor:

-- Takip icin bekleyen siparisleri bul
SELECT order_id, customer_id, order_date, total_amount
FROM orders
WHERE status = 'pending'
ORDER BY order_date;

Veri dagilimi: Siparislerin yalnizca %2'si status = 'pending' durumunda (2.000.000'den 40.000).

Gorev: Bu sorgu icin en verimli indeksi tasarlayin.

Optimal Cozum: Kismi indeks olusturun:

-- PostgreSQL kismi indeks
CREATE INDEX idx_orders_pending_date
ON orders (order_date)
WHERE status = 'pending';

Faydalar:

Metrik Tam Indeks Kismi Indeks
Indekslenen satirlar 2.000.000 40.000 (%2)
Indeks boyutu (yaklasik) ~60 MB ~1.2 MB
Insert yuku Her insert Yalnizca bekleyen siparisler

SQL Server esdegeri (Filtered Index):

CREATE INDEX idx_orders_pending_date
ON orders (order_date)
WHERE status = 'pending';

Alistirma 2.3: Kapsayici Indeks Tasarimi

Bu sorgu kontrol panelinizde dakikada binlerce kez calistirilir:

SELECT order_date, total_amount
FROM orders
WHERE customer_id = 12345
ORDER BY order_date DESC
LIMIT 10;

Gorev: Bu sorgunun tamamen indeksten yanit alinmasini saglayan bir indeks tasarlayin (Index-Only Scan).

Cozum - Kapsayici Indeks:

-- PostgreSQL (anahtar olmayan sutunlari dahil et)
CREATE INDEX idx_orders_customer_covering
ON orders (customer_id, order_date DESC)
INCLUDE (total_amount);

-- Alternatif: Tum sutunlar indeks anahtarinda
CREATE INDEX idx_orders_customer_covering_alt
ON orders (customer_id, order_date DESC, total_amount);

Neden calisiyor:

  • customer_id - Esitlik filtresi icin ilk anahtar
  • order_date DESC - Siralama icin ikinci anahtar (zaten sirada!)
  • total_amount - Tablo erisimi gerekmemesi icin dahil edilmis
Index Only Scan using idx_orders_customer_covering
  (cost=0.43..12.56 rows=10 width=16)
  Index Cond: (customer_id = 12345)
  Heap Fetches: 0  <-- Tablo erisimi yok!
INCLUDE vs Anahtar Sutunlar:
Yalnizca SELECT'te gereken sutunlar icin INCLUDE kullanin (WHERE/ORDER BY'da kullanilmayan). Bu, indeks agacini daha kucuk ve hizli tutar.

Alistirma 2.4: Ifade Indeksi

Kullanicilar e-posta ile musteri arar, ancak arama buyuk/kucuk harf duyarsizdir:

-- Mevcut yavas sorgu
SELECT * FROM customers
WHERE LOWER(email) = 'john.doe@example.com';

email uzerindeki normal bir indeks, LOWER() fonksiyonu nedeniyle yardimci olmaz.

Gorev: Bu buyuk/kucuk harf duyarsiz aramayi destekleyen bir indeks olusturun.

Cozum - Ifade Indeksi:

-- PostgreSQL ifade indeksi
CREATE INDEX idx_customers_email_lower
ON customers (LOWER(email));

-- Sorgu ayni ifadeyi kullanmalidir:
SELECT * FROM customers
WHERE LOWER(email) = LOWER('John.Doe@Example.com');

Alternatif yaklasimlar:

-- PostgreSQL: Buyuk/kucuk harf duyarsiz metin icin citext uzantisi kullanin
CREATE EXTENSION IF NOT EXISTS citext;
ALTER TABLE customers ALTER COLUMN email TYPE citext;
-- Simdi normal indeks calisiyor: WHERE email = 'John.Doe@Example.com'

-- SQL Server: Buyuk/kucuk harf duyarsiz collation kullanin
CREATE INDEX idx_customers_email ON customers (email);
-- Varsayilan collation genellikle buyuk/kucuk harf duyarsizdir
Onemli: Sorgu, indeksle TAMAMEN ayni ifadeyi kullanmalidir. WHERE UPPER(email) = ... bu indeksi KULLANMAZ!

Bolum 3: Performans Icin Sorgu Yeniden Yazma

Yavas sorgulari optimize edilmis versiyonlara donusturun.

Alistirma 3.1: Indeksli Sutunlarda Fonksiyonlardan Kacinma

Bu sorgu, order_date uzerinde indeks olmasina ragmen yavas calisiyor:

-- Yavas sorgu (indeks kullanilamaz)
SELECT * FROM orders
WHERE YEAR(order_date) = 2024;

Gorev: order_date uzerindeki indeksi kullanmak icin sorguyu yeniden yazin.

Optimize Edilmis Sorgu:

-- Fonksiyon yerine aralik kosulu kullanin
SELECT * FROM orders
WHERE order_date >= '2024-01-01'
  AND order_date < '2025-01-01';
Once (Yavas): 
Seq Scan on orders
cost=0.00..55123.00
Filter: YEAR(order_date) = 2024
 Sonra (Hizli): 
Index Scan using idx_orders_date
cost=0.43..8456.00
Index Cond: (order_date >= ... AND ...)

Daha fazla ornek:

-- Kotu: Sutunda fonksiyon
WHERE DATE(created_at) = '2024-06-15'
-- Iyi: Aralik kosulu
WHERE created_at >= '2024-06-15' AND created_at < '2024-06-16'

-- Kotu: Sutunda hesaplama
WHERE total_amount * 1.1 > 100
-- Iyi: Hesaplamayi sabite tasiyin
WHERE total_amount > 100 / 1.1

-- Kotu: String fonksiyonu
WHERE SUBSTRING(code, 1, 3) = 'ABC'
-- Iyi: Onek ile LIKE kullanin
WHERE code LIKE 'ABC%'

Alistirma 3.2: Iliskili Alt Sorguyu JOIN ile Degistirme

Bu sorgu, musterilerin en son siparislerini bulur, ancak cok yavas calisiyor:

-- Yavas: Iliskili alt sorgu (her musteri icin bir kez calisir)
SELECT
    c.customer_id,
    c.email,
    (
        SELECT MAX(o.order_date)
        FROM orders o
        WHERE o.customer_id = c.customer_id
    ) AS last_order_date
FROM customers c
WHERE c.is_active = TRUE;

Gorev: Daha iyi performans icin JOIN kullanarak yeniden yazin.

JOIN Kullanilarak Optimize Edilmis Sorgu:

-- Hizli: JOIN ile tek toplama
SELECT
    c.customer_id,
    c.email,
    MAX(o.order_date) AS last_order_date
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.is_active = TRUE
GROUP BY c.customer_id, c.email;

Window Function Kullanan Alternatif (tam siparis detaylarini alin):

-- Gercek en son siparisi alin, sadece tarihi degil
WITH ranked_orders AS (
    SELECT
        o.*,
        ROW_NUMBER() OVER (
            PARTITION BY o.customer_id
            ORDER BY o.order_date DESC
        ) AS rn
    FROM orders o
)
SELECT
    c.customer_id,
    c.email,
    ro.order_date AS last_order_date,
    ro.total_amount AS last_order_amount
FROM customers c
LEFT JOIN ranked_orders ro
    ON c.customer_id = ro.customer_id AND ro.rn = 1
WHERE c.is_active = TRUE;
Performans Karsilastirmasi:
  • Iliskili alt sorgu: O(musteriler x siparisler) - HER musteri icin alt sorgu calistirir
  • JOIN yaklasimi: O(musteriler + siparisler) - her tablodan tek gecis

Alistirma 3.3: EXISTS vs IN vs JOIN

En az bir siparis vermis tum musterileri bulun. Uc yaklasimi karsilastirin:

-- Yaklasim 1: IN alt sorgusu
SELECT * FROM customers
WHERE customer_id IN (SELECT customer_id FROM orders);

-- Yaklasim 2: EXISTS alt sorgusu
SELECT * FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id);

-- Yaklasim 3: DISTINCT ile JOIN
SELECT DISTINCT c.* FROM customers c
JOIN orders o ON c.customer_id = o.customer_id;

Gorev: Hangi yaklaşimin en iyi oldugunu ve ne zaman oldugunu analiz edin.

Analiz:

Yaklasim En Iyi Durum Performans
EXISTS Buyuk orders tablosu, sadece varlik kontrolu Ilk esleşmede durur (erken sonlandirma)
IN Kucuk alt sorgu sonuc kumesi Once tam listeyi olusturmalidir
JOIN + DISTINCT Her iki tablodan sutunlara ihtiyac duyuldugunda Tekrar kaldirma icin ekstra is

Bu ozel durum icin oneri:

-- EXISTS genellikle varlik kontrolleri icin en iyisidir
SELECT * FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.customer_id
);

Tum yaklasimlar icin gerekli indeks:

CREATE INDEX idx_orders_customer_id ON orders (customer_id);
Modern Optimize Ediciler: Cogu modern veritabani (PostgreSQL 12+, SQL Server 2019+) genellikle IN ve EXISTS'i benzer planlar uretecek sekilde optimize edebilir. Her zaman EXPLAIN ile dogrulayin.

Alistirma 3.4: UNION ALL vs UNION

Aktif ve VIP musterileri birlestirmeniz gerekiyor (VIP ayri bir bayraktir):

-- Mevcut yavas sorgu
SELECT customer_id, email, 'active' AS source
FROM customers WHERE is_active = TRUE
UNION
SELECT customer_id, email, 'vip' AS source
FROM customers WHERE is_vip = TRUE;

Gorev: Ayni musterinin hem aktif hem de VIP olabilecegini bilerek bu sorguyu optimize edin (ve bu durumda her iki satiri da gormek istiyorsunuz).

Optimize Edilmis Sorgu:

-- Tekrarlar kasitli oldugu icin UNION ALL kullanin
SELECT customer_id, email, 'active' AS source
FROM customers WHERE is_active = TRUE
UNION ALL
SELECT customer_id, email, 'vip' AS source
FROM customers WHERE is_vip = TRUE;
UNION (Yavas):
  • Sonuclari birlestirir
  • Tum satirlari siralar
  • Tekrarlari kaldirir
 UNION ALL (Hizli):
  • Sonuclari birlestirir
  • Tamamlandi!
  •  
UNION (ALL degil) ne zaman kullanilir:
UNION'i yalnizca ozellikle tekrarlari kaldirmaniz gerektiginde ve sorgular ayni satirlari dondurebileceginde kullanin. Sorgular karsilikli olarak ozel ise veya tum satirlari istiyorsaniz, UNION ALL kullanin.

Bolum 4: Zorluk Alistirmalari

Birden fazla optimizasyon teknigini birlestiren karmasik senaryolar.

Zorluk 4.1: Bir Raporlama Sorgusunu Optimize Etme

Bu aylik satis raporu 45 saniye calisiyor. Optimize edin:

-- Mevcut yavas sorgu
SELECT
    p.category,
    DATE_TRUNC('month', o.order_date) AS month,
    COUNT(DISTINCT o.order_id) AS order_count,
    SUM(oi.quantity * oi.unit_price) AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
WHERE o.order_date >= '2024-01-01'
  AND o.status IN ('completed', 'shipped')
GROUP BY p.category, DATE_TRUNC('month', o.order_date)
ORDER BY month, revenue DESC;

Gorev: Yurutme suresini azaltmak icin optimizasyonlari (indeksler, sorgu yeniden yazmalari) tanimlayin.

Adim 1: Destekleyici Indeksler Olusturun

-- Siparis filtreleme ve birlestirme icin indeks
CREATE INDEX idx_orders_date_status
ON orders (order_date, status)
WHERE status IN ('completed', 'shipped');

-- order_items join icin indeks
CREATE INDEX idx_order_items_order_id
ON order_items (order_id)
INCLUDE (product_id, quantity, unit_price);

-- products join icin indeks
CREATE INDEX idx_products_id_category
ON products (product_id)
INCLUDE (category);

Adim 2: Sorgu Optimizasyonu

-- Daha iyi plan icin CTE kullanan optimize edilmis sorgu
WITH filtered_orders AS (
    SELECT order_id, order_date
    FROM orders
    WHERE order_date >= '2024-01-01'
      AND status IN ('completed', 'shipped')
)
SELECT
    p.category,
    DATE_TRUNC('month', fo.order_date) AS month,
    COUNT(DISTINCT fo.order_id) AS order_count,
    SUM(oi.quantity * oi.unit_price) AS revenue
FROM filtered_orders fo
JOIN order_items oi ON fo.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
GROUP BY p.category, DATE_TRUNC('month', fo.order_date)
ORDER BY month, revenue DESC;

Adim 3: Sik Raporlar Icin Materialized View Dusunun

-- Bu rapor sik calisiyorsa, onceden hesaplayin
CREATE MATERIALIZED VIEW mv_monthly_sales AS
SELECT
    p.category,
    DATE_TRUNC('month', o.order_date) AS month,
    COUNT(DISTINCT o.order_id) AS order_count,
    SUM(oi.quantity * oi.unit_price) AS revenue
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
WHERE o.status IN ('completed', 'shipped')
GROUP BY p.category, DATE_TRUNC('month', o.order_date);

-- Gunluk yenile
REFRESH MATERIALIZED VIEW mv_monthly_sales;
Beklenen Iyilestirme: Indekslerle 45 saniye -> 2-5 saniye, veya materialized view ile saniyenin altinda.

Zorluk 4.2: Sayfalama Performansi

Bu sayfalama sorgusu, kullanicilar sonraki sayfalara gittikce yavaslıyor:

-- Buyuk offset'ler icin yavas (ornegin, sayfa 1000)
SELECT order_id, order_date, total_amount
FROM orders
WHERE customer_id = 12345
ORDER BY order_date DESC
LIMIT 20 OFFSET 19980;  -- Sayfa 1000 x sayfa basina 20

Gorev: Alternatif olarak keyset (cursor tabanli) sayfalama uygulayin.

OFFSET'in Sorunu:

  • Veritabani 20 dondurmeden once 19.980 satiri taramali ve atmalidir
  • Performans sayfa numarasiyla dogrusal olarak duser

Cozum: Keyset Sayfalama

-- Ilk sayfa
SELECT order_id, order_date, total_amount
FROM orders
WHERE customer_id = 12345
ORDER BY order_date DESC, order_id DESC
LIMIT 20;

-- Sonraki sayfalar: son satirin degerlerini cursor olarak kullanin
-- Son satir: order_date='2024-03-15', order_id=98765 ise
SELECT order_id, order_date, total_amount
FROM orders
WHERE customer_id = 12345
  AND (order_date, order_id) < ('2024-03-15', 98765)
ORDER BY order_date DESC, order_id DESC
LIMIT 20;

Destekleyici Indeks:

CREATE INDEX idx_orders_customer_date_id
ON orders (customer_id, order_date DESC, order_id DESC);
Metrik OFFSET (Sayfa 1000) Keyset
Taranan satirlar ~20.000 ~20
Yurutme suresi ~500ms ~1ms
Tutarlilik Veri degisirse kacirma/coklatma olabilir Stabil cursor konumu
Takas: Keyset sayfalama dogrudan "sayfa N'ye git"i desteklemez. Sonsuz kaydirma veya "daha fazla yukle" kaliplari icin en iyisidir.

Zorluk 4.3: Sorgu Plani Analizi

  AND o.order_date >= '2024-01-01'

Optimize Edilmis Sorgu:

-- Gerekli indeksler olusturulduktan sonra
SELECT c.email, COUNT(o.order_id), SUM(o.total_amount)
FROM customers c
INNER JOIN orders o ON c.customer_id = o.customer_id
WHERE c.email LIKE '%@gmail.com'  -- veya trigram indeksi kullanin
  AND o.order_date >= '2024-01-01'
GROUP BY c.email
HAVING SUM(o.total_amount) > 1000;
